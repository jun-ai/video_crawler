# Vocabulary.vue 优化建议报告

**文件**：`/root/video_crawler/frontend/src/views/Vocabulary.vue`（1097 行）
**审查日期**：2026-06-18
**审查范围**：UI 布局 / 功能完整性 / 后端 API 利用度

---

## 0. 当前能力速览

### 前端功能
- 生词列表（分页 10/20/50/100）、按语料来源过滤、5 种排序
- 掌握状态三档筛选（全部 / 学习中 / 已掌握）
- 中文显示开关（PageHeader 上的 SfSwitch，无 label）
- 每张卡片：单词 + 音标 + TTS 朗读 + 翻译 + 掌握度环 + 复习强度条 + 语境 + 复习次数 + 来源 + 时间
- 操作：查询发音、标记掌握、移出、删除
- 空状态、加载状态、移动端响应式（768/480 两档断点）

### 后端已提供但前端 Vocabulary.vue 未使用的 API
| API | 用途 | 当前使用方 |
|---|---|---|
| `GET /learning/vocabulary/review-queue` | 拉取待复习生词（SM-2 队列） | VocabularyReview.vue |
| `POST /learning/vocabulary/review` | 提交复习评分 | VocabularyReview.vue |
| `GET /learning/vocabulary/review-stats` | 复习统计（due/learning/mastered） | VocabularyReview.vue |
| `GET /learning/vocabulary/lookup` | 单词释义查询（已用） | ✓ |
| 字段 `next_review_at` / `last_reviewed_at` / `ease_factor` / `interval_days` | 间隔重复元数据 | **完全未在前端展示** |

> 核心发现：后端已投入 SM-2 间隔重复算法的全部基建，但 Vocabulary.vue 主列表页完全没接进来，"何时该复习"这个产品价值点对用户是隐形的。

## P0 — 必须做（影响核心使用）

### P0-1 「移出」和「删除」是同一个接口，写了两个名字

**问题**：`Vocabulary.vue:201-216` 给"已掌握"词显示「移出」，给"学习中"词显示「删除」，但两者都调用 `vocabularyAPI.delete(id)`（`learning.py:299`）。语义混淆——"移出"暗示还有恢复机制，实际物理删除。

**改什么**：
- 二选一：要么去掉「移出」按钮（已掌握 = 留在生词本里只是变灰），要么让"移出"变成软删（加个 `archived` 字段）。
- 建议方案：已掌握项只显示「取消掌握」（调 `markMastered` 反向接口），删除按钮永远叫「删除」，文案统一。

**工作量**：小（1-2h）
**影响面**：高频——每个用户都至少会操作几十次。

---

### P0-2 整张卡片有三个 TTS 触发点，互相打架

**问题**：
- 行 85：`vocab-word-row @click="speakText(item.word)"` —— 点单词朗读
- 行 95：`tts-btn @click="speakText(item.word)"` —— 显式 TTS 圆形按钮
- 行 188：「查询发音」按钮调 `lookupAndSpeak` —— 也是朗读

三个入口、两种行为（普通朗读 vs 查询后再朗读），用户分不清「查询发音」和点耳机图标有何区别。

**改什么**：
- 把 44px 圆形按钮保留为唯一显式 TTS 入口。
- 单词本身可点击，但 hover 提示"朗读"；移除单独的「查询发音」按钮（或把它合并成"长按查询释义"）。
- 「查询发音」按钮改名「查释义」，并明确语义：未缓存时走 lookup API（耗时），已缓存时直接朗读。

**工作量**：小（1h）
**影响面**：高频，每个生词至少点 1 次。

---

### P0-3 完全缺搜索框

**问题**：用户存了 200+ 词后，找一个特定词只能滚动 + 翻页。无搜索入口。

**改什么**：
- filter-bar 加搜索框（防抖 300ms），前端模糊匹配 `word` 字段（不需要后端改）。或者后端加 `?keyword=xxx`，简单 LIKE 查询。
- 搜索词保留在 URL query 里（刷新/分享状态）。

**工作量**：中（后端 1h + 前端 2h）
**影响面**：高频——词越多越痛。

---

### P0-4 完全缺批量操作

**问题**：100 条「学习中」需要全部删除或全部标记掌握时，必须一条条点 100 次。

**改什么**：
- 卡片左上角加 checkbox + 顶部出现「已选 N 项」操作栏（批量标记掌握 / 批量删除）。
- 后端补 `POST /learning/vocabulary/batch` 接收 `ids: []` + `action: 'master'|'delete'`。

**工作量**：中（后端 2h + 前端 3h）
**影响面**：中——词到 50+ 才有感，但用户黏性来自这种省力操作。

---

### P0-5 主页面没有复习入口

**问题**：`review-queue` API 已上线、`VocabularyReview.vue` 已写完，但生词本主页没有"待复习 N 词，开始复习 →"的引导。

**改什么**：
- filter-bar 顶部或 PageHeader 加一个醒目的 `amber-gold` 强调卡片："今日待复习 8 个词 · 开始复习 →"
- 点击跳 `/vocabulary/review`。
- 数据来自 `vocabularyAPI.getReviewStats()`。

**工作量**：小（2h，前端）
**影响面**：高——复习是产品的留存核心场景。

---

### P0-6 中文开关无标签

**问题**：`Vocabulary.vue:6-9` 的 `<SfSwitch v-model="showChinese" />` 没有 label / icon。新用户完全不知道这开关控制什么。

**改什么**：
- 改成 `<SfSwitch v-model="showChinese">显示中文</SfSwitch>`，或在 PageHeader actions 里把开关包成带文字的 toggle。
- 同时把状态持久化到 localStorage（行 284 `ref(true)` 每次重置）。

**工作量**：小（0.5h）
**影响面**：中（用户体验摩擦）。

## P1 — 应该做（体验提升明显）

### P1-1 掌握度进度环和复习强度条完全重复

**问题**：`Vocabulary.vue:122-152` 同时渲染了：
- `mastery-ring`：SVG 环显示 `review_count × 15%`
- `review-strength`：横向 bar 显示**同一个** `review_count × 15%`

数据相同，仅形式不同。视觉上让卡片"重量翻倍"，用户也困惑。

**改什么**：二选一。建议**保留进度环**（更精致），删掉 bar；或者改成显示**「预计下次复习：2 天后」**这种 SM-2 才有价值的信息（用 `next_review_at` 字段）。

**工作量**：小（1h）
**影响面**：每张卡片视觉减肥。

---

### P1-2 完全没暴露 SM-2 间隔重复元数据

**问题**：后端 Vocabulary 模型（`learning.py:1390-1463`）已实现完整的 SM-2 算法（`next_review_at` / `ease_factor` / `interval_days`），但 Vocabulary.vue 卡片上**完全没展示这些字段**。用户看不到"我下次什么时候该复习这个词"。

**改什么**：
- 卡片加一行小字："下次复习：明天"（按 `next_review_at` 距今天数）。
- 已到期的词标红/橙（amber-gold），提示用户优先复习。
- 后端 list 接口响应里把 `next_review_at` / `last_reviewed_at` 一起返回（现在没返回，见 `learning.py:250-261`）。

**工作量**：中（后端加字段 0.5h + 前端展示 2h）
**影响面**：高——把"会复习"产品故事可视化。

---

### P1-3 缺少"新词"独立筛选

**问题**：当前 filterMastered 三个 chip 是「全部 / 学习中 / 已掌握」（行 47-64），但 `vocab-new`（未复习过的新词）和 `vocab-learning`（已复习但未掌握）被合并到"学习中"里。用户想集中复习**从未见过的**新词时无法筛选。

**改什么**：增加第四个 chip「待首次复习」，或拆为「未复习 N / 复习中 M / 已掌握 K」三档。后端加 `reviewed: bool` 过滤参数。

**工作量**：中（后端 1h + 前端 1h）
**影响面**：中。

---

### P1-4 缺少导出功能

**问题**：生词本数据无出口，无法导出到 Anki / CSV / JSON。

**改什么**：
- 后端补 `GET /learning/vocabulary/export?format=csv|json|anki`。
- 前端在 PageHeader 加「导出」按钮，下拉选格式。
- Anki 格式：每行 `<word>\t<phonetic>\t<translation>\t<context>\t<material_title>`，UTF-8 BOM。

**工作量**：中（后端 3h + 前端 1h）
**影响面**：中（迁移/备份需求）。

---

### P1-5 卡片信息密度过高

**问题**：单卡片 ~7 个区块（状态条 + 词头 + 音标 + TTS + 翻译 + 进度环 + 强度条 + 语境 + 来源 + 时间 + 操作），10 张卡片就是一屏 + 半。

**改什么**：
- 默认收起「语境中文」和「进度可视化」（折叠面板 / "展开详情"按钮）。
- 列表模式 vs 卡片模式切换：列表模式只显示 [单词] [音标] [复习次数] [状态] [操作]，密度 ×3。
- 移动端默认只显示 4 个核心字段。

**工作量**：中（3h）
**影响面**：高（每个用户每天打开 1+ 次）。

---

### P1-6 删除操作无 Undo

**问题**：`Vocabulary.vue:430-441` 删除走 confirm，但确认后立即物理删除，toast 只说"已删除"。误删无解。

**改什么**：toast 改为带「撤销」按钮（5 秒内可恢复）。后端删除可先标 `deleted_at` 软删，或前端本地缓存最近删除的 id 列表。

**工作量**：中（3h）
**影响面**：中（防误操作）。

---

### P1-7 排序"复习最多"和卡片自带的复习次数 tag 重复信息

**问题**：`Vocabulary.vue:38` 排序选项"复习最多"，但每张卡片的 `vocab-status-group`（行 104-110）已经在显示 `复习 N 次`。这个排序价值不大。

**改什么**：把"复习最多"换成更有价值的排序，比如「最久未复习」（按 `last_reviewed_at` ASC）或「最该复习」（按 `next_review_at` ASC，先到期）。后端 order_map 加这两个分支（`learning.py:232-238`）。

**工作量**：小（1h）
**影响面**：低-中。

## P2 — 锦上添花（可后续迭代）

### P2-1 wordInfoCache 存了 `example` 字段但永远不渲染

`Vocabulary.vue:346` 缓存时存了 `result.example`，但模板里从来不用。死字段。建议：要么删掉、要么在卡片里以"例句"形式渲染。

**工作量**：小（0.5h）

---

### P2-2 单词查询失败完全静默

`Vocabulary.vue:348-350` catch 后只 console.error。卡片永远显示无音标无翻译，用户不知道是没缓存还是 API 挂了。改：失败时卡片对应字段显示 "?" + tooltip 说明。

**工作量**：小（1h）

---

### P2-3 语料筛选下拉一次拉 100 条且不可搜索

`Vocabulary.vue:301-307` `getList({ page_size: 100 })`。50 个语料以上的用户下拉很难用。

**改什么**：SfSelect 改为支持搜索的 Combobox，或者下拉分组"最近学习 / 全部"。

**工作量**：中（3h）

---

### P2-4 列表入场无动画

切换 filter / 翻页时，v-for 直接重渲染，无 stagger。视觉显得突兀。

**改什么**：用 `<TransitionGroup>` + 每张卡片 `transition-delay: index * 30ms`。

**工作量**：小（1h）

---

### P2-5 单词无收藏 / 星标

已掌握和未掌握是个粗粒度二分法，但用户常想"这个词重要但不熟"。建议加 `starred` 字段，前端用 amber-gold 星星图标。

**工作量**：中（后端 + 前端 4h）

---

### P2-6 无键盘快捷键

复习场景高频操作，j/k 上下切换、m 标记掌握、d 删除这种快捷键能极大提升效率。

**工作量**：小（2h）

---

### P2-7 无无限滚动选项

分页 OK 但复习场景无限滚动更顺手。增加"虚拟列表 / 分页"切换。

**工作量**：中（4h）

---

### P2-8 showChinese 状态不持久

行 284 每次进入页面重置为 `true`。个人偏好应该跨会话保留。

**工作量**：小（0.5h）

---

### P2-9 el-pagination 与 Sf 设计系统不一致

行 247-254 用了 Element UI 的分页，但整个项目已经迁移到 Sf 系列组件。视觉割裂。

**改什么**：写一个 `SfPagination.vue` 套一层；或接受临时不一致。

**工作量**：中（3-4h）

---

### P2-10 单词重复添加无提示

如果同一个词已经在生词本里，再点"加入生词"会创建重复条目（取决于后端逻辑，前端 `vocabularyAPI.add` 没去重检查）。建议加入前先调 `getList` 检查。

**工作量**：小（1h）

## 建议执行顺序（单开发者）

1. **第 1 周**：P0-6 → P0-1 → P0-2 → P1-1（一组 UI 小修，~1 天）
2. **第 2 周**：P0-5（接复习入口）+ P1-2（暴露 SM-2 数据）—— 把产品核心价值可视化
3. **第 3 周**：P0-3（搜索）→ P1-5（列表模式）
4. **第 4 周**：P0-4（批量操作）+ P1-4（导出）—— 偏 power user 功能
5. **后续**：P1-3 / P1-6 / P2 系列小迭代

## 关键洞察

- **后端 SM-2 算法完全被埋没**：产品已投入间隔重复基建，但 Vocabulary.vue 主页面 0 展示，这是最大的产品价值 gap。**P1-2 + P0-5 是收益最高的组合拳**。
- **冗余 UI 集中在卡片右下**：三 TTS 入口 + 双进度可视化 + 移出/删除同义，是设计冗余的重灾区。
- **批量 + 搜索 + 导出** 三件套对个人开发者来说价值不高，但对留存至关重要（Anki 等竞品的基本盘）。
